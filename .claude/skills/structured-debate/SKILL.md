# 構造化ディベートスキル

## このスキルの目的

議論の質を高め、根拠に基づいた結論を必ず出せ。「一方で〜もあります」で終わる両論併記を禁止する。

## なぜ構造化ディベートが必要か

- **AIは合意バイアスを持つ**: AIは対立を避け、「どちらも一理あります」で逃げがち。これでは意思決定に使えない
- **表面的な反論は価値がない**: 「リスクもあります」だけでは何も判断できない。具体的にどのくらいのリスクがあるかを定量的に示す必要がある
- **議論なしの合意は危険**: 全員（全エージェント）が同意した計画でも、検討不足で失敗することがある

## 構造化ディベートの基本ルール

### ルール1: 命題を明確にする

議論を始める前に、命題をYes/Noで答えられる形に変換せよ。

```
❌ 悪い命題: 「AIトレーディングについて議論しよう」
→ 焦点がなく、何を決めるかが不明

✅ 良い命題: 「LLMによるニュースセンチメント分析を
  売買シグナルに組み込むべきか？」
→ Yes/Noで判断でき、根拠を整理しやすい

✅ さらに良い命題: 「LLMによるニュースセンチメント分析を
  売買シグナルに組み込むことで、
  バックテストのシャープレシオが0.3以上改善するか？」
→ 定量的な基準があり、検証可能
```

### ルール2: 賛成/反対の構造

```
1. 賛成側: 根拠を3つ提示（各根拠にデータ・事例を添える）
2. 反対側: 各根拠に対する反論を3つ提示
3. 賛成側: 再反論
4. 反対側: 最終反論
5. 判定: 根拠の強さで結論を出す（多数決ではない）
```

### ルール3: 根拠の質を担保する

根拠には以下のランクがある。上位の根拠が優先される：

| ランク | 種類 | 例 |
|--------|------|-----|
| S | 定量データ（自分の検証結果） | バックテストでシャープレシオが1.2→1.5に改善 |
| A | 定量データ（公開研究） | 学術論文でセンチメント分析が年率3%のα |
| B | 定性的だが具体的な根拠 | 決算発表後のGAP-UPの70%が初日に形成される |
| C | 一般論・理論的推論 | 感情分析は理論的に有効であるはず |
| D | 直感・経験則 | なんとなく良さそう |

**Cランク以下の根拠のみで結論を出してはならない。**

### ルール4: 結論を必ず出す

```
❌ 禁止パターン:
「賛成側にも反対側にも一理あります。
状況に応じて判断してください。」
→ 何も決まっていない。価値ゼロ。

✅ 必須パターン:
「結論: [採用/不採用/条件付き採用]
理由: [最も決定的だった根拠]
条件: [もし〜ならば見直す]
次のアクション: [具体的に何をするか]」
```

## AIエージェント特有の問題と対策

### 問題1: 合意バイアス

AIは反論を避け、先に出た意見に同調しがち。

**対策: DEVILロールの設計**

議論では必ず1つのエージェント（またはロール）を「DEVIL（悪魔の代弁者）」に指名する。DEVILのルール：

1. **必ず反対意見を述べる**: たとえ賛成でも、反対の立場で議論する
2. **最低3つの反論を出す**: 「〜かもしれない」ではなく、具体的なシナリオで反論
3. **最悪のケースを提示する**: 楽観バイアスを打ち破る

```
✅ DEVILの良い反論:
「このセンチメント分析戦略は、2022年3月のFRB利上げ局面で
テストしていない。あの時期、ニュースのトーンはネガティブだったが
市場は「bad news is good news」で上昇した。
つまりセンチメントの極性が反転する局面が存在し、
その検出メカニズムがない限り、この戦略は危険だ」

❌ DEVILの悪い反論:
「一方で、この戦略にはリスクもあるかもしれません」
→ 具体性ゼロ。表面的な反論は禁止。
```

### 問題2: 「両論併記で終わる」問題

**対策: 判定プロトコル**

議論の最後に必ず以下のフォーマットで判定を出す：

```markdown
## 判定

### 命題
[Yes/Noの命題]

### 結論: [Yes / No / 条件付きYes]

### 決定的な根拠（ランク付き）
1. [S/A/B/C] [根拠の内容]
2. [S/A/B/C] [根拠の内容]

### 却下された反論と理由
1. [反論] → [なぜ却下されたか]

### 残存リスク
- [対処できていないリスクとその影響度]

### 次のアクション
1. [具体的なタスク]
2. [具体的なタスク]

### 見直し条件
- [何が起きたらこの判断を見直すか]
```

### 問題3: 表面的な反論

**対策: 反論品質チェックリスト**

反論を出す前に、以下を満たしているか確認せよ：

- [ ] 具体的なシナリオ・数字が含まれているか
- [ ] 「〜かもしれない」ではなく「〜の場合に〜が起きる」の形か
- [ ] 反論の根拠のランクはB以上か
- [ ] 対案（こうすべき）が含まれているか

## 議論の品質チェックリスト

議論終了時に以下をすべて確認せよ：

- [ ] 命題がYes/Noの形に明確化されているか
- [ ] 賛成側が根拠3つ以上を提示したか
- [ ] 反対側（DEVIL含む）が具体的な反論を提示したか
- [ ] 根拠にランクが付与されているか
- [ ] 結論が明確に出ているか（両論併記で終わっていないか）
- [ ] 次のアクションが定義されているか
- [ ] 見直し条件が設定されているか
