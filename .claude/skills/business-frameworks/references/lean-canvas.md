# リーンキャンバス リファレンス

## 概要

リーンキャンバスはAsh Mauryaが「Running Lean」で提唱した1ページのビジネスモデル記述ツール。ビジネスモデルキャンバス（Osterwalder）をスタートアップ向けに改良したもの。

**なぜリーンキャンバスを使うか**: 長い企画書を書く前に、アイデアの核心を1ページで検証可能な形にまとめるため。

## 9つのセルと書き方

### 1. 課題（Problem）

上位3つの課題を特定せよ。

```
❌ 悪い例:
- 株式投資は難しい
- お金を増やしたい

✅ 良い例:
- 個人投資家は決算発表後の急激な株価変動に
  対応できない（情報処理速度の限界）
- 感情的バイアス（損切りできない、利確が早すぎる）
  により期待値がマイナスになる
- 複数のニュースソースを統合的に判断する
  認知負荷が高すぎる
```

**書き方のコツ**:
- 「誰が」「どんな状況で」「何に困っているか」を明記する
- 既存の代替手段も書く（課題が本当に存在するなら、人々は何かで代替しているはず）

### 2. 顧客セグメント（Customer Segments）

ターゲットユーザーを特定せよ。まずアーリーアダプターから。

```
トレーディングボットの場合:
- 主要ユーザー: 自分自身（個人投資家、プログラミング可能）
- アーリーアダプター: まず自分の資金で検証
- 将来の拡大: 同じようなプログラマー投資家
```

### 3. 独自の価値提案（Unique Value Proposition）

なぜ既存のソリューションではなくこれを使うべきかを一文で。

```
❌ 悪い例: 「AIを使った高度なトレーディングシステム」
→ 差別化ポイントが不明

✅ 良い例: 「LLMのニュース理解力と定量分析を組み合わせ、
決算発表後30分以内に投資判断を自動化する
個人投資家向けトレーディングエージェント」
→ 何が（LLM+定量）、いつ（決算後30分）、誰に（個人投資家）
```

### 4. ソリューション（Solution）

各課題に対するソリューションを対応させよ。

```
課題1（情報処理速度）→ Claude APIで決算レポートを自動解析
課題2（感情バイアス）→ ルールベースの売買判断で感情を排除
課題3（認知負荷）→ マルチソース情報を統合スコアに集約
```

### 5. チャネル（Channels）

ユーザーにどうリーチするか。個人プロジェクトの場合は自分自身が唯一のユーザーなので簡略化可能。

### 6. 収益の流れ（Revenue Streams）

```
トレーディングボットの場合:
- 直接収益: トレーディングによるキャピタルゲイン
- 間接収益: 知見の蓄積（将来のサービス化可能性）
- コスト構造との関係: 収益がAPI費用等を上回る必要あり
```

### 7. コスト構造（Cost Structure）

```
固定費:
- Alpaca Market Data: $0（Basic）/ $9-99/月（Pro/Ultra）
- Claude API: 従量課金（月$10-50想定）
- サーバー費用: $0（ローカル実行）/ $5-20（クラウド）

変動費:
- 取引手数料: $0（Alpacaは手数料無料）
- スリッページ: 注文サイズ依存
```

### 8. 主要指標（Key Metrics）

何を測定すれば成功/失敗が判断できるか。

```
✅ 良い例:
- シャープレシオ > 1.0（リスク調整後リターン）
- 月次リターン > S&P500（ベンチマーク超過）
- 最大ドローダウン < 15%
- 勝率 × 平均利益 > 敗率 × 平均損失（正の期待値）
```

### 9. 圧倒的優位性（Unfair Advantage）

簡単にコピーできない優位性。正直に書くこと。

```
❌ 悪い例: 「AIを使っている」
→ 誰でもできる

✅ 良い例: 「LLMの自然言語理解を売買判断に統合する
パイプラインの設計知見。この知見は試行錯誤の中で
蓄積されるもので、コードだけコピーしても再現困難」
→ 経験に基づく暗黙知は簡単にコピーできない
```
