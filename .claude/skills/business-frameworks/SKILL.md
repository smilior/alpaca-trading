# ビジネスフレームワーク適用スキル

## このスキルの目的

企画・戦略立案時に適切なフレームワークを選択し、正しく適用せよ。フレームワークは「考える道具」であり、「埋めるフォーム」ではない。

## なぜフレームワークが必要か

- **思考の抜け漏れを防ぐ**: フレームワークなしでは、自分の得意な領域しか考えない
- **チーム間の共通言語**: AIエージェント同士が議論する際、構造化された形式で意見を交換できる
- **意思決定の質を上げる**: 直感だけでなく、体系的な分析に基づく判断ができる

## 使えるフレームワーク一覧と使い分け

| フレームワーク | 使うタイミング | 詳細 |
|---|---|---|
| リーンキャンバス | 新しいプロダクト・戦略のアイデアを整理するとき | [references/lean-canvas.md](references/lean-canvas.md) |
| なぜなぜ分析（5 Whys） | 問題の根本原因を特定するとき | [references/five-whys.md](references/five-whys.md) |
| 事前検死（Pre-mortem） | 計画の実行前にリスクを洗い出すとき | [references/pre-mortem.md](references/pre-mortem.md) |
| SWOT / PEST分析 | 外部環境と内部能力を整理するとき | [references/swot-pest.md](references/swot-pest.md) |

### 選択フローチャート

```
何をしたい？
├── 新しいアイデアを整理したい → リーンキャンバス
├── 問題が起きた。原因を知りたい → なぜなぜ分析
├── これから計画を実行する。失敗が怖い → 事前検死
└── 現状を俯瞰的に分析したい → SWOT + PEST
```

## フレームワーク適用のルール

### 必ず守ること

1. **「So What?」テスト**: 各セル・各分析結果に対して「だから何？」と問え。アクションに繋がらない分析は無価値
2. **具体性の確保**: 「市場が大きい」ではなく「米国株市場の日次取引高は約5000億ドル」のように数字で語れ
3. **反証可能性**: 「AIトレーディングは有望」ではなく「RSI+センチメント分析で月次シャープレシオ1.5を達成できる」のように検証可能な形にせよ

### アンチパターン（やってはいけないこと）

#### 悪い例：形だけ埋める

```
【リーンキャンバスの「課題」セル】
❌ 悪い例: 「株式投資で利益を出すのが難しい」
→ 漠然としすぎ。誰のどんな困難かが不明

✅ 良い例: 「個人投資家の90%が短期トレードで損失を出す。
   主因は①感情的な売買判断、②ニュース反応の遅れ、
   ③リスク管理の欠如。特に②は機械化で解決可能」
→ 具体的で、解決の方向性が見える
```

#### 悪い例：深掘りしない

```
【SWOT分析の「強み」】
❌ 悪い例: 「AIを活用できる」
→ 具体的に何のAIでどう強みになるか不明

✅ 良い例: 「Claude APIによる自然言語処理で、
   決算レポート・ニュース・SNSを統合的に分析し、
   人間のアナリストが8時間かける作業を5分で実行可能。
   特に決算発表直後の30分間の情報優位性が大きい」
→ 何がどう強みになるか、時間軸も含めて具体的
```

#### 悪い例：複数フレームワークの結果を統合しない

```
❌ 悪い例: SWOT分析とリーンキャンバスを別々に作って終わり
✅ 良い例: SWOT分析の結果をリーンキャンバスの
   「課題」「ソリューション」「優位性」に反映させる
```

## 「良い企画」と「悪い企画」の判定基準

### 良い企画の条件（すべて満たすこと）

1. **具体的な仮説がある**: 「〜すれば〜になる」の形で述べられる
2. **検証方法が明確**: どうやって仮説を確認するかが決まっている
3. **失敗条件が定義されている**: 何をもって「この企画は失敗」と判断するか
4. **撤退基準がある**: どこまで損失が出たら止めるか
5. **リソース見積もりがある**: 時間・コスト・必要なデータの見積もり

### 悪い企画の兆候

- 「AIで株を売買して儲ける」のような漠然とした目標しかない
- 成功するシナリオしか考えていない
- 「とりあえずやってみよう」で検証計画がない
- 競合（他のアルゴリズムトレーダー）の分析がない
- リスク管理の話が一切出てこない

## AIエージェントチームでの使い方

複数のAIエージェントが企画を議論する場合、以下の手順を踏め：

1. **企画担当エージェント**: リーンキャンバスを作成し、提示する
2. **批評担当エージェント**: なぜなぜ分析で前提を疑い、事前検死で失敗シナリオを洗い出す
3. **統合担当エージェント**: 両者の出力をSWOT分析で整理し、最終的なGo/No-Go判定を下す

この構造により、「全員が賛成して失敗する」リスクを低減できる。
